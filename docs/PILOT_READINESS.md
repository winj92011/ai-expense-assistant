# AI 报销助手试点成熟度

本文用于判断当前原型离“可发布演示”和“可小范围试点”还有多远，并给后续开发排优先级。

## 当前阶段

当前版本约处在 **55% 产品化成熟度**：

- 对外演示、内部讲解、投资人 Demo：约 80% 可用。
- 小范围真实业务试点：约 40% 可用。
- 企业生产长期使用：约 20-25% 可用。

这意味着当前版本已经适合演示完整产品故事，但还不应承诺真实企业长期运行。

## 已经稳定的能力

- 员工提交、主管审批、财务复核、出纳付款的报销闭环。
- Kimi AI 票据识别接入，并保留 mock 兜底。
- base 出发地选择、闭环行程判断、行程完整性建议。
- 财务归档号、入账状态、凭证包、审计说明、导出台账 CSV。
- 飞书、钉钉、浏览器测试平台选择。
- 飞书/钉钉 mock 消息卡片回调体验。
- 统一 `currentUser` 身份上下文层。
- 前端 state 到接近数据库对象结构的映射。
- 数据对象 JSON 导出。
- 原型控制台：平台、身份、回调、数据模型收纳展示。
- 第一层英文界面。
- Playwright E2E、合同检查、页面内 smoke 验证。

## 仍属于原型的部分

- 数据主要保存在浏览器和前端状态中，不是真实企业数据库。
- 角色切换是原型模拟，不是正式登录权限。
- 飞书/钉钉回调是页面内 mock，不是真实平台 webhook。
- 票据附件还没有接对象存储。
- 财务入账和付款没有对接真实财务系统或银行系统。
- 审计日志还没有不可篡改、可追溯的服务端落库。

## 从 55% 到 70% 的优先事项

目标：进入“可小范围试点原型”。

### 1. 真实持久化最小版本

必须保存用户、部门、报销单、报销明细、票据附件元数据、审批任务、审计日志和财务归档。

建议先接一个轻量数据库，不急着做复杂企业级架构。只要能做到刷新页面、换设备后仍能恢复报销单，就能显著提高试点可信度。

### 2. 真实身份上下文最小版本

先不做复杂权限系统，先完成统一身份入口：

- 浏览器测试用户。
- 飞书用户。
- 钉钉用户。
- 后端统一解析成 `currentUser`。

角色仍可先用配置或 mock 映射，但用户身份应由服务端返回。

### 3. 票据附件存储

最小版本只需要上传文件、保存文件 URL 或 storage key、让报销明细关联附件，并在凭证包中引用附件。

### 4. 服务端审计链

每个关键动作都要保存：谁操作、在哪个平台操作、操作了哪张单据、操作前后状态、时间、回调 payload 或页面动作 payload。

## 70% 试点验收标准

- 用户提交的报销单可持久化保存。
- 刷新页面后仍能看到单据。
- 换角色或重新打开页面后仍能继续处理同一张单据。
- 附件元数据可保存。
- 审批和财务动作写入审计日志。
- 自动测试覆盖核心闭环。
- 线上 smoke 稳定通过。
- 文档清楚说明哪些是真实能力，哪些仍是 mock。

## 暂不进入的范围

- 复杂多租户计费。
- 银行付款自动化。
- 财务系统深度集成。
- 完整国际化。
- 复杂企业组织同步。
- 正式安全合规认证。

这些能力可以在 70% 之后再分阶段推进。

## 当前建议路线

1. 先把当前演示版文档收口。
2. 设计数据库表和最小 API。
3. 让报销单、明细、审批任务先能真实保存。
4. 再把身份上下文从前端 mock 改为服务端返回。
5. 最后接附件存储和审计日志。

完成前 3 步后，产品就会从“演示原型”明显进入“可试点系统”。
